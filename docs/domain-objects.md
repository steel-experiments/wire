# Domain Objects

Wire uses seven core domain objects. They are conceptually separate and must remain separate in code and persistence.

## Entity ID system

All entity IDs are prefixed strings with the format `{prefix}_{random}`:

| Prefix | Type | Example |
|--------|------|---------|
| `task_` | `TaskId` | `task_abc123` |
| `run_` | `RunId` | `run_def456` |
| `session_` | `SessionId` | `session_ghi789` |
| `skill_` | `SkillId` | `skill_jkl012` |
| `artifact_` | `ArtifactId` | `artifact_mno345` |
| `action_` | `ActionId` | `action_pqr678` |
| `approval_` | `ApprovalId` | `approval_stu901` |
| `event_` | `TraceEventId` | `event_vwx234` |
| `policy_` | `PolicyDecisionId` | `policy_yza567` |
| `hypothesis_` | `HypothesisId` | `hypothesis_bcd890` |
| `experiment_` | `ExperimentId` | `experiment_efg123` |
| `profile_` | `ProfileId` | `profile_hij456` |
| `comparison_` | `ComparisonId` | `comparison_klm789` |

Generated by `createId()` in `src/shared/ids.ts`.

## Task

A user objective. The top-level unit of work.

```ts
interface Task {
  id: TaskId;
  title: string;
  mode: "task" | "investigate" | "experiment";
  objective: string;
  constraints: string[];
  successCriteria: string[];
  falsificationCriteria?: string[];
  budget?: TaskBudget;
  createdAt: string;
}
```

| Field | Description |
|-------|-------------|
| `mode` | `task` for practical work, `investigate` for diagnosis, `experiment` for research branching |
| `constraints` | Things the agent must not do |
| `successCriteria` | What "done" looks like |
| `budget` | Optional limits: max runs, tokens, browser minutes, USD |

Modes:
- **task** — deterministic work (fill forms, download files, extract data)
- **investigate** — diagnosis and debugging (where does the flow fail?)
- **experiment** — research with branching (try multiple strategies, compare outcomes)

## Run

One execution attempt of a task. A task can have multiple runs (branches in experiment mode).

```ts
interface Run {
  id: RunId;
  taskId: TaskId;
  parentRunId?: RunId;         // set when this run is a branch
  branchLabel?: string;         // e.g. "branch-2"
  hypothesisId?: HypothesisId;  // linked hypothesis in experiment mode
  status: RunStatus;
  startedAt?: string;
  finishedAt?: string;
  result?: string;
  outcomeSummary?: string;
  classification?: RunClassification;
}
```

| Status | Meaning |
|--------|---------|
| `queued` | Waiting to start |
| `running` | Agent loop is active |
| `awaiting-approval` | Paused for human approval |
| `succeeded` | Completed (may or may not have met criteria) |
| `failed` | Terminated with errors |
| `aborted` | User cancelled |

### Run classification

Every run gets a classification at finalization:

| Kind | Meaning |
|------|---------|
| `task-complete` | Success criteria met with evidence |
| `partial-success` | Some criteria met, some failed |
| `blocked-auth` | Hit a login or captcha wall |
| `site-error` | The site failed (429, 500, etc.) |
| `agent-error` | The agent made errors |
| `infra-error` | Network, browser crash, or infrastructure failure |
| `counterexample` | Experiment found a counterexample |
| `ambiguous` | Insufficient evidence to classify |

Classification logic lives in `src/agent/classify.ts` and considers: code success/failure counts, artifact presence, objective relevance, contract checks, artifact reviews, error patterns, auth walls, and budget exhaustion.

## Session

A live remote Chrome instance managed by Steel.

```ts
interface BrowserSession {
  id: SessionId;
  provider: "steel" | "custom";
  profileId?: ProfileId;
  liveUrl?: string;         // URL for live viewing
  debugUrl?: string;        // URL for debug access
  wsUrl?: string;           // WebSocket endpoint
  createdAt: string;
  status: SessionStatus;
  region?: string;
  proxyCountryCode?: string | null;
}
```

One session per run. Created at run start, stopped at run end (unless `keepSessionOpen` is set).

## Skill

Durable site knowledge stored as markdown with YAML frontmatter.

```ts
interface SkillMetadata {
  id: SkillId;
  scope: "domain" | "workflow" | "interaction";
  status?: SkillStatus;
  hostnamePatterns?: string[];
  tags: string[];
  updatedAt: string;
  source: "builtin" | "team" | "generated";
  confidence?: number;
  sourceRunIds?: RunId[];
  supersedes?: SkillId[];
}
```

Skills contain:
- Durable site shape (URL patterns, API endpoints)
- Stable selectors
- Known traps and failure modes
- Wait patterns with reasons

Skills must **not** contain: secrets, tokens, cookies, run transcripts, chain-of-thought, or step-by-step narration.

See [skills-system.md](skills-system.md) for the full skill format and lifecycle.

## Artifact

Persisted evidence from a run.

```ts
interface Artifact {
  id: ArtifactId;
  runId: RunId;
  kind: ArtifactKind;
  path: string;
  mimeType?: string;
  createdAt: string;
  metadata?: JsonObject;
}
```

Artifact kinds: `screenshot`, `html`, `markdown`, `pdf`, `download`, `helper-diff`, `skill-proposal`, `json-output`, `plot`, `table`, `note`, or any custom string.

Artifacts are persisted to disk and their metadata is saved as JSON. Content is deduplicated via blob storage.

## PolicyDecision

The result of a policy engine evaluation.

```ts
interface PolicyDecision {
  id: PolicyDecisionId;
  actionId: ActionId;
  result: "allow" | "deny" | "require-approval";
  reason?: string;
}
```

- **allow** — action proceeds
- **deny** — action is blocked (deletions, irreversible mutations)
- **require-approval** — action pauses the run for human review

See [policy-engine.md](policy-engine.md) for the full rules.

## TraceEvent

Immutable observability record. Every significant action during a run produces a trace event.

```ts
interface TraceEvent {
  id: TraceEventId;
  runId: RunId;
  ts: string;
  kind: TraceEventKind;
  payload: JsonObject;
}
```

Event kinds:

| Kind | When it fires |
|------|---------------|
| `thought-summary` | Agent reasoning, stop reasons, pause/resume |
| `observation` | Page orientation snapshot |
| `code-exec` | JavaScript sent to the browser |
| `code-result` | JavaScript execution result |
| `artifact` | Artifact recorded (extraction, note, summary) |
| `policy-check` | Policy engine evaluation |
| `approval-request` | Approval requested for a gated action |
| `approval-result` | Approval resolution |
| `skill-load` | Skills loaded for the current context |
| `skill-empty` | Skill directory has no files |
| `skill-proposal` | LLM-proposed skill after run completion |
| `contract-check` | Completion contract validation |
| `artifact-review` | LLM artifact quality review |
| `llm-call` | LLM API call (when `--trace-llm` is enabled) |
| `llm-usage` | Token usage per LLM call |
| `user-message` | User message injected during a running task |
| `error` | Execution errors |

Trace events are append-only. They are never mutated after creation.
