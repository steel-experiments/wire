# Wire by Steel

## Tagline

The zero-weight browser agent.

## Positioning

Wire is a thin agent runtime for real browser work.

It gives agents a small, code-first way to operate persistent cloud browsers through Steel, produce evidence, learn from failures, and preserve reusable site knowledge without becoming a heavy browser framework.

## Core Idea

Steel carries the browser weight.
Wire carries intent through it.

The name should feel thin, strong, conductive, and minimal: a direct line between agent intent and real web execution.

## What Wire Is

- A zero-weight browser agent.
- A code-first runtime for real web tasks.
- A research workbench for browser runs, failures, traces, and experiments.
- A way to turn repeated site discoveries into durable skills.
- A small core with clear policy, trace, session, profile, and skill boundaries.

## What Wire Is Not

- Not a giant browser framework.
- Not screenshot-only RPA.
- Not an opaque self-healing box.
- Not prompt soup.
- Not a swarm system.
- Not a replacement for Steel browser infrastructure.

## Message Pillars

### Real Browser Work

Wire operates against real remote Chrome sessions: auth state, JavaScript, files, dialogs, downloads, latency, blockers, and all.

### Zero-Weight Core

The runtime stays small. Browser infrastructure belongs to Steel. Durable knowledge belongs in files. Policy stays outside model reasoning.

### Code-First Actions

Agents act through inspectable code, not a swollen tool vocabulary. Helpers are allowed, but every abstraction needs an escape hatch.

### Evidence By Default

Every run should leave proof: observations, executed code, artifacts, traces, failures, classifications, and comparisons where useful.

### Learning Per Run

Wire values diagnosis, counterexamples, and better next experiments as much as happy-path completion.
